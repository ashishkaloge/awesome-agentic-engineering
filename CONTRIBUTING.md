# Contributing

Contributions are welcome when they make the list more accurate, useful, and
safe for public engineering use.

## Good Contributions

- A reputable public tool, workflow, prompt pattern, MCP server, skill, plugin, or evaluation method.
- A concise description that explains the engineering use case.
- A public link to official documentation, a maintained repository, or a credible project page.
- A correction that removes stale, broken, duplicated, or low-signal links.

## Safety Rules

- Do not submit private project files, logs, exports, generated dumps, binaries, or archives.
- Do not submit secrets, credentials, tokens, environment files, or production configuration.
- Do not submit leaked prompts, cracked tools, license bypasses, malware, spam tooling, scraping abuse, or bulk messaging workflows.
- Do not submit links that depend on private access, invite-only downloads, or suspicious mirrors.

## Awesome List Format

Use this format for each item:

```markdown
- [Name](https://example.com/) - Short description that starts with a capital letter and ends with a period.
```

Keep descriptions neutral, specific, and short. Prefer official documentation or
the canonical source repository.

## Local Checks

Run the Awesome List linter before opening a pull request:

```bash
npx --yes awesome-lint
```

## Pull Requests

Keep pull requests focused. A small PR that adds one excellent resource is
better than a broad rewrite with weak links.
